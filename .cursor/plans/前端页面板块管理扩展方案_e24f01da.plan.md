# 前端页面板块管理 — 全功能扩展方案

## 一、设计原则（大厂思维）

1. **页面即配置**：前端每一个可视板块都对应后台一个可管理的实体
2. **所见即可管**：运营人员无需改代码即可调整首页文案、图片、链接、排序
3. **类型化存储**：每种板块类型有独立的 Prisma 模型（而非全塞进 Setting JSON），确保类型安全、可查询、可排序
4. **继承现有模式**：遵循已有的 AdSlot/Ad 两级模式（容器->条目），复用 AdminUI 组件体系
5. **渐进接入**：先建后台管理能力，前端逐步从硬编码切换到动态数据，互不阻塞

## 二、新增 Prisma 模型

```prisma
// ——— 页面板块管理 ———

enum SectionType {
  HERO_SLIDE       // 首页Hero轮播（多张图+文案+按钮）
  TOPIC_CARD       // 专题卡片（IC设计/制造等）
  TESTIMONIAL      // 声音引语
  NAV_LINK         // 主导航链接
  FOOTER_COLUMN    // 页脚列（含多个链接）
  CTA_SECTION      // 首页合作CTA区
}

model PageSection {
  id        String      @id @default(cuid())
  type      SectionType // 板块类型
  code      String      @unique // 程序查找码，如 "hero", "topics", "testimonials", "main-nav", "footer", "cta"
  title     String?     // 后台显示名称
  active    Boolean     @default(true)
  sort      Int         @default(0)
  createdAt DateTime    @default(now())
  updatedAt DateTime    @updatedAt

  items     PageSectionItem[]
  settings  Json?       // 该板块的额外配置（如Hero的autoplay间隔、CTA的背景色等）

  @@index([type, active])
}

model PageSectionItem {
  id          String         @id @default(cuid())
  sectionId   String
  section     PageSection    @relation(fields: [sectionId], references: [id], onDelete: Cascade)
  title       String?        // 标题
  subtitle    String?        // 副标题/摘要/kicker
  description String?        // 详细描述
  image       String?        // 图片URL
  link        String?        // 链接URL
  linkText    String?        // 按钮文字
  sort        Int            @default(0)
  active      Boolean        @default(true)
  startDate   DateTime?
  endDate     DateTime?
  extra       Json?          // 扩展字段（如Hero slide的alt文本、页脚列的header文本等）
  createdAt   DateTime       @default(now())
  updatedAt   DateTime       @updatedAt

  @@index([sectionId, sort])
}
```

## 三、各板块映射关系


| 前端板块     | 前端文件                                 | PageSection.code | PageSection.type | 条目用途                                  |
| -------- | ------------------------------------ | ---------------- | ---------------- | ------------------------------------- |
| Hero 区   | `home-hero.tsx`                      | `"hero"`         | `HERO_SLIDE`     | 每个条目=一页轮播（图片+标题+副标题+CTA按钮文字+链接）       |
| 专题入口     | `home-topics.tsx` + `home-topics.ts` | `"topics"`       | `TOPIC_CARD`     | 每个条目=一个专题卡片（标题+描述+链接）                 |
| 声音引语     | `home-proof.tsx`                     | `"testimonials"` | `TESTIMONIAL`    | 每个条目=一条引语（内容+作者属性）                    |
| CTA 区    | `home-cta.tsx`                       | `"cta"`          | `CTA_SECTION`    | 一个条目=kicker+标题+副标题+按钮1文本/链接+按钮2文本/链接  |
| 主导航      | `site-header.tsx`                    | `"main-nav"`     | `NAV_LINK`       | 每个条目=一个导航项（标签+链接+排序）                  |
| 页脚       | `site-footer.tsx`                    | `"footer"`       | `FOOTER_COLUMN`  | 每个条目=一列（列标题+extra.header），再展开其子条目作为链接 |
| 广告Banner | `home-ad-banner.tsx`                 | 已有AdSlot/Ad系统    | —                | 不迁移，保持现有广告管理                          |

## 六、关键设计决策

1. **轮播 vs 单图 Hero**：Hero 设计为多图轮播（`code: "hero"` 板块下多个 items），支持运营上传多张宣传图，前端实现自动轮播+Pagination Dots。每张 slide 独立配置标题、副标题、CTA 按钮文字和链接。
2. **图片管理**：沿用当前系统的图片URL录入方式（不实现本地文件上传），Hero slides 和 Topic cards 的 `image` 字段填入图片链接。如需本地上传可复用已有 Upload API。
3. **Footer 列的子链接**：利用 `extra: Json` 字段存储页脚列的子链接数组（`[{label: "资讯", href: "/articles"}, ...]`），无需额外建表。
4. **向后兼容**：前端组件改造时保留 fallback 逻辑——如果查询不到动态数据，回退到当前硬编码值，确保数据迁移不中断。
