# 芯师爷官网 — 技术架构方案

## 一、架构设计原则

### 1.1 大厂架构思维

芯师爷官网的技术架构遵循以下核心原则，这些原则来自一线互联网大厂的工程实践：

| 原则 | 说明 | 落地方式 |
|------|------|---------|
| **关注点分离** | 展示层、业务层、数据层严格分离 | App Router 的 Server/Client 组件 + Service Layer |
| **约定优于配置** | 减少决策疲劳，统一团队规范 | ESLint + Prettier + commitlint 全链路代码规范 |
| **防御性编程** | 所有外部输入不可信 | Zod 全链路运行时验证 |
| **渐进增强** | 核心功能离线可用 | Service Worker + App Shell 架构 |
| **优雅降级** | 高版本功能不影响低版本体验 | 特性检测 + polyfill 策略 |
| **可观测性** | 系统状态可度量、可追踪 | OpenTelemetry + 结构化日志 |

## 二、技术选型详解

### 2.1 前端层

| 技术 | 版本 | 选型理由 | 核心用法 |
|------|------|---------|---------|
| **Next.js** | 16.x | 成熟的全栈框架，App Router 支持 RSC、Server Actions | SSR 前台页面、API Route、中间件 |
| **React** | 19.x | 组件化 UI 构建，Concurrent 模式 | Server/Client 组件分层 |
| **TypeScript** | 5.x | 全栈类型安全 | strict 模式，类型共享前后端 |
| **Tailwind CSS** | 4.x | 原子化 CSS，零运行时，设计 token 系统 | 设计系统 + 响应式 |
| **Zustand** | 5.x | 轻量状态管理，无 boilerplate | 客户端复杂交互状态 |
| **@formatjs/intl** | 3.x | ICU 消息格式，行业标准 | 中英文国际化 |
| **React Aria** | — | 无头 UI 组件库，内置无障碍 | 表单、弹窗、列表等交互组件 |
| **next-intl** | — | Next.js 国际化最佳实践 | 路由国际化、消息管理 |

### 2.2 后端层

| 技术 | 版本 | 选型理由 | 核心用法 |
|------|------|---------|---------|
| **Next.js Route Handlers** | 16.x | 无需额外框架，天然共享类型 | API 暴露 |
| **Server Actions** | 16.x | 表单提交和可变操作，类型安全 | 企业后台 CRUD |
| **Prisma** | 7.x | 类型安全的 ORM，迁移自动化 | 数据模型 + 查询 |
| **Zod** | 4.x | 运行时类型验证 | API 入参校验、表单验证 |
| **JWT + bcryptjs** | — | 无状态认证，无需 Session 存储 | 用户认证 |
| **Upstash Redis** | — | Serverless Redis，按量计费 | 缓存、限流、队列 |
| **Meilisearch** | — | 开箱即用的中文全文搜索 | 企业/产品/文章搜索 |

### 2.3 基础设施层

| 技术 | 选型理由 |
|------|---------|
| **Vercel** | 原生 Next.js 部署，Edge Network，ISR 支持 |
| **Neon (Serverless Postgres)** | 按需扩缩，分支数据库，适合开发/生产环境隔离 |
| **Cloudflare R2** | S3 兼容，零出口费，适合图片/文件存储 |
| **Upstash Redis** | Serverless 架构，无需维护 Redis 集群 |
| **Resend** | 开发者友好的邮件发送服务 |
| **Sentry** | 全栈错误追踪和性能监控 |
| **Plausible / Umami** | 隐私优先的网站分析，无需 Cookie 弹窗 |

## 三、核心架构决策

### 3.1 渲染策略

根据页面类型选择不同的渲染模式：

| 页面类型 | 渲染模式 | 原因 |
|---------|---------|------|
| 首页、企业列表、文章列表 | ISR (60s) | 内容有时效性，SEO 优先 |
| 企业详情页 | ISR (300s) | 企业信息更新不频繁 |
| 文章详情页 | ISR (600s) | 发布后很少修改 |
| 企业后台、管理后台 | CSR + SSR fallback | 需要实时数据，客户端交互密集 |
| 搜索结果页 | SSR (每次请求) | 依赖用户输入，无法预渲染 |
| 评选结果页 | ISR (30s) | 投票数据实时变化 |

### 3.2 Server Component 与 Client Component 边界

```
Public Pages (SSR/ISR)
├── Layout (Server Component)
│   ├── Header (Server → 静态导航)
│   │   └── SearchBar (Client → 交互)
│   ├── Page Content (Server → 数据获取)
│   │   └── Interactive elements (Client → 事件)
│   └── Footer (Server → 静态)

Enterprise Dashboard (CSR + SSR fallback)
├── Layout (Server)
│   ├── Sidebar (Server → 导航结构)
│   ├── Dashboard Content (Client → 实时数据)
│   └── DataTable (Client → 排序/筛选/分页)
```

### 3.3 认证与权限设计

```
认证流程：
┌─────────┐     ┌──────────┐     ┌──────────┐
│ 登录请求  │ ──→ │ 验证凭据   │ ──→ │ 签发 JWT  │
└─────────┘     └──────────┘     └──────────┘
                                        │
                                        ↓
┌─────────┐     ┌──────────┐     ┌──────────┐
│ 受保护路由 │ ←── │ 中间件验证  │ ←── │ Cookie   │
│ 或 API   │     │ (Edge)   │     │ HttpOnly│
└─────────┘     └──────────┘     └──────────┘

权限模型：
- 访客：可读公共内容
- 注册用户：可投票、投递简历
- 企业用户：管理自己企业内容
- 运营编辑：管理内容、审核企业
- 管理员：全平台管理
```

### 3.4 API 设计规范

```
格式：/api/{version}/{resource}/{id}[/action]

示例：
  GET    /api/v1/companies          → 企业列表
  GET    /api/v1/companies/[id]     → 企业详情
  POST   /api/v1/companies          → 创建企业
  PATCH  /api/v1/companies/[id]     → 更新企业
  DELETE /api/v1/companies/[id]     → 删除企业

约定：
  - 统一响应格式：{ data, meta, error }
  - 分页：{ page, pageSize, total, totalPages }
  - 错误：{ code, message, details }
  - 筛选：查询参数 q（搜索）、filter（筛选）、sort（排序）
```

### 3.5 搜索架构

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 数据写入   │ ──→ │ 消息队列   │ ──→ │ Meilisearch│
```

## 五、性能优化策略

### 5.1 前端性能

- **图片优化**：Next.js Image 组件 + WebP/AVIF 格式 + 响应式图片
- **代码分割**：动态导入（`next/dynamic`）非首屏组件
- **字体优化**：`next/font` 自托管字体，消除 Google Fonts 网络请求
- **缓存策略**：ISR + CDN 缓存 + 浏览器缓存（`stale-while-revalidate`）
- **Bundle 分析**：持续监控 Bundle 大小，设定阈值告警

### 5.2 后端性能

- **数据库**：Prisma 查询优化（select 白名单、include 按需、batch 操作）
- **缓存**：Redis 缓存热门列表数据、API 响应缓存
- **Connection Pool**：Prisma 连接池配置，Serverless 环境适配
- **懒加载**：列表页分页 + 无限滚动

### 5.3 前端工程化

```
代码规范链：
  ESLint (规则检查)
    → Prettier (自动格式化)
    → Husky (Git Hooks)
    → lint-staged (增量检查)
    → commitlint (提交信息规范)
    → CI (PR 自动检查)
```
