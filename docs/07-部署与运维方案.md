# 芯师爷官网 — 部署与运维方案

## 一、环境架构

### 1.1 三环境策略

```
                     Git Branch
                         │
                    ┌────┴────┐
                    │         │
                  main     feature/*
                    │         │
                    │     feature branch deploy
                    │     (Vercel Preview)
                    │
              ┌─────┴─────┐
              │            │
           staging       production
        (vercel.app)   (xinshiye.com)
```

| 环境 | 域名 | 用途 | 数据库 | 部署方式 |
|------|------|------|--------|---------|
| **Development** | localhost:3000 | 本地开发 | 本地 PostgreSQL | `npm run dev` |
| **Staging** | xinshiye-staging.vercel.app | 集成测试、验收 | Neon (分支) | PR 自动部署 |
| **Production** | xinshiye.com (或自定义域名) | 生产运行 | Neon (主库) | main 分支自动部署 |

## 二、部署流程

### 2.1 CI/CD 管线 (GitHub Actions)

```yaml
# .github/workflows/ci.yml
name: CI
on: [pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test
        ports:
          - 5432:5432
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx prisma migrate deploy
      - run: npm run test
      - run: npm run test:e2e

# .github/workflows/cd.yml
name: Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}
          vercel-args: '--prod'
```

### 2.2 CI 门禁标准

```
PR 合并到 main 的条件：
  [x] Lint 通过
  [x] TypeScript 类型检查通过
  [x] 单元测试通过
  [x] E2E 测试通过
  [x] 至少 1 人 Code Review Approve
  [x] Lighthouse 分数 > 90 (可选门禁)
```

### 2.3 Vercel 配置

**项目配置**：
- Framework Preset: Next.js
- Build Command: `npm run build`
- Output Directory: `.next`
- Node.js Version: 22.x
- Environment Variables: 通过 Vercel Dashboard 管理

**Environment Variables**：

| 变量名 | 说明 | Staging | Production |
|--------|------|---------|------------|
| `DATABASE_URL` | PostgreSQL 连接串 | Neon Staging DB | Neon Production DB |
| `REDIS_URL` | Redis 连接串 | Upstash Staging | Upstash Production |
| `JWT_SECRET` | JWT 签名密钥 | 随机字符串 | 随机字符串 |
| `NEXT_PUBLIC_SITE_URL` | 站点 URL | staging 域名 | 正式域名 |
| `R2_ACCESS_KEY` | R2 访问密钥 | ✓ | ✓ |
| `R2_SECRET_KEY` | R2 密钥 | ✓ | ✓ |
| `R2_BUCKET_NAME` | R2 存储桶 | xinshiye-staging | xinshiye |
| `MEILISEARCH_URL` | 搜索服务地址 | ✓ | ✓ |
| `MEILISEARCH_KEY` | 搜索服务密钥 | ✓ | ✓ |
| `SENTRY_DSN` | Sentry DSN | ✓ | ✓ |

## 三、数据库运维

### 3.1 Neon (Serverless PostgreSQL)

**优势**：
- Serverless 按需计费，冷启动 < 500ms
- 分支数据库：每个 PR 自动创建数据库分支
- 快照备份：每日自动备份，保留 7 天

**迁移策略**：

```bash
# 开发
npx prisma migrate dev --name add_company_table

# Staging
npx prisma migrate deploy

# Production
npx prisma migrate deploy
# 大表迁移：先创建 shadow 数据库，分批次迁移
```

### 3.2 备份策略

| 备份类型 | 频率 | 保留周期 | 恢复目标 (RTO) |
|---------|------|---------|---------------|
| 自动快照 | 每日 | 7 天 | < 1 小时 |
| WAL 归档 | 实时 | 14 天 | 时间点恢复 (PITR) |
| 手动快照 | 更新前 | 永久 | 立即恢复 |

## 四、监控与告警

### 4.1 监控体系

```
三层监控：
  1. 基础设施层  → 数据库连接、Redis 命中率、CDN 状态
  2. 应用层      → API 响应时间、错误率、吞吐量
  3. 业务层      → 企业入驻数、文章发布量、广告收入
```

### 4.2 Sentry 配置

```typescript
// src/lib/sentry.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.1,     // 性能追踪采样率
  profilesSampleRate: 0.1,   // Profiling 采样率
  replaysSessionSampleRate: 0.1,
  replaysOnErrorSampleRate: 1.0,

  // 忽略非关键错误
  ignoreErrors: [
    'NetworkError',
    'AbortError',
  ],
});
```

### 4.3 告警规则

| 告警项 | 阈值 | 通知方式 |
|--------|------|---------|
| 5xx 错误率 | > 1% | 即时 Slack / 飞书 |
| API P95 响应时间 | > 1s | 即时 Slack |
| 数据库连接数 | > 80% | Slack |
| Lighthouse 分数 | < 85 | 日报 |
| 磁盘/存储 | > 80% | Slack |
| 企业入驻数异常 | 日环比下降 > 50% | Slack |

## 五、日志管理

### 5.1 结构化日志

```typescript
// src/lib/logger.ts
const logger = pino({
  level: process.env.NODE_ENV === 'production' ? 'info' : 'debug',
  formatters: {
    level(label) { return { level: label }; },
  },
  serializers: {
    req: (req) => ({
      method: req.method,
      url: req.url,
      userId: req.user?.id,
    }),
```

```typescript
const securityHeaders = [
```

## 八、灾备与恢复

### 8.1 故障恢复流程

```
1. 发现问题 → Sentry 告警 / 用户反馈
2. 评估影响 → 确定 P0/P1/P2 级别
3. 回滚方案 → Vercel Rollback (一键回滚到上一版本)
4. 修复上线 → Hotfix 分支 → PR → 合并 main → 自动部署
5. 事后复盘 → 5 Whys 分析 → 改进措施 → 更新 Runbook
```

### 8.2 回滚策略

```bash
# Vercel Dashboard 一键回滚
# 或 CLI 回滚
vercel rollback --yes

# 数据库回滚
npx prisma migrate down 1

# 注意：数据库回滚可能导致数据丢失，
# 生产环境优先选择"向前修复"而非"回滚"
```
